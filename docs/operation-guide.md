# 運用ガイド

生駒祭 2026 情宣AI判定システムの運用フローをまとめる。

## 前提条件

- 利用者: `@kindai.ac.jp` の Google アカウント保持者のみ
- プラットフォーム: Web アプリ（PWA）、PC・スマホ対応
- Supabase: 生駒祭統合管理の共有プロジェクト（`qlldsvpkcfftbibujltf`）上で稼働

## 1. 事前準備（祭り前）

管理者が以下の作業を実施する。

### 1.1 権限設定

共有 Supabase の `roles` テーブルに情宣用 screen 権限を設定する。

| ロール | 追加する screen |
|--------|---------------|
| 広報部 | `josenai_review_koho` |
| 企画管理部 | `josenai_review_kikaku` |
| 管理者 | `josenai_review_koho`, `josenai_review_kikaku`, `josenai_admin` |

### 1.2 マスタデータ登録

マスタ管理画面から CSV をインポートする。

**団体 CSV:**
```csv
organization_code,organization_name,category
ORG001,文化会演劇部,文化会
ORG002,軽音楽部,文化会
ORG003,写真部,文化会
```

**企画 CSV:**
```csv
project_code,project_name,organization_code
PRJ001,秋公演「夢の続き」,ORG001
PRJ002,ライブステージ,ORG002
PRJ003,写真展「光と影」,ORG003
```

### 1.3 ルール・ガイドライン登録

ルール管理画面から以下の文書を登録・編集する。

| 文書種別 | 内容 |
|---------|------|
| 情宣ルール | 禁止事項、必須記載事項、審査フロー |
| 著作物取り扱いガイドライン | 画像・フォント・音楽・二次創作の取り扱い |
| 提出ガイド | ファイル形式、サイズ制限、リスクスコアの目安 |

### 1.4 システム設定

設定画面で以下を確認・調整する。

| 設定項目 | デフォルト値 | 説明 |
|---------|------------|------|
| sandbox_daily_limit | 3 | サンドボックス1日上限回数 |
| submission_enabled | true | 提出受付中フラグ |
| ai_timeout_seconds | 30 | AI判定タイムアウト秒数 |
| auto_approve_enabled | false | 自動承認機能の有効/無効 |
| auto_approve_threshold | 10 | 自動承認閾値（この値以下のスコアを自動承認） |
| 各管理者パスワード | (bcrypt ハッシュ) | 広報部・企画管理部・管理者の認証用 |

### 1.5 自動承認機能の免責事項

自動承認機能を有効化する前に、管理者は以下を確認すること。

**管理者の確認事項:**
- 自動承認された提出物に関する責任は、機能を有効化した管理者およびその所属組織（大学祭実行委員会）が負う
- システム開発者・AI サービス提供者（Google LLC）は、自動承認の結果について責任を負わない
- AI のリスクスコア算出は確率的な推論に基づいており、判定精度は 100% ではない
- 自動承認された提出物は定期的にサンプリング確認することを推奨する
- 問題が発見された場合、管理者は速やかに承認を取り消し（`rejected` に変更）できる

**有効化手順:**
1. 設定画面で「自動承認」トグルを ON にする
2. 免責確認モーダルが表示される
3. 内容を確認し「有効化する」をクリック
4. 必要に応じて閾値を調整する（デフォルト: 10%）

## 2. 出展団体（一般ユーザー）の利用フロー

### 2.1 ログイン

1. ログイン画面で [Google でログイン] をクリック
2. `@kindai.ac.jp` の Google アカウントで認証
3. 管理者パスワードモーダルで [スキップ] を選択
4. 初回ログイン時、`user_profiles` と `josenai_profiles` が自動作成される

### 2.2 サンドボックス（事前確認）

正式提出前に AI 判定を試行できる（1日3回まで）。

1. 提出先を選択（企画管理部 / 広報部）
2. 団体名をプルダウンから選択
3. 企画名をプルダウンから選択（選択した団体に紐づく企画のみ表示）
4. 媒体種別を選択（ポスター A1, ビラ, SNS画像 等）
5. ファイルをアップロード
6. [AI判定を実行] をクリック
7. リスクスコアと指摘内容が表示される

**特記事項:**
- データは保存されない（プライバシー保護）
- リスクが高い場合は修正して再試行を推奨

### 2.3 正式提出

1. サンドボックスと同じフォームに入力
2. AI 判定が実行される
3. リスクレベルに応じた提出フロー:

| リスクレベル | スコア | 動作 |
|-------------|--------|------|
| 低リスク | 0-10% | ワンタップで提出完了 |
| 中リスク | 11-50% | 警告表示、確認後に提出 |
| 高リスク | 51-100% | 理由入力が必須 |

4. 提出完了後:
   - ファイルが Google Drive に保存される
   - `josenai_submissions` にレコードが作成される
   - 自動承認が有効な場合、以下の条件に応じてステータスが決定される:

| 条件 | 結果 | ステータス |
|------|------|-----------|
| 自動承認 OFF | 手動審査キューへ | `pending` |
| 自動承認 ON + AI スキップ | 手動審査キューへ | `pending` |
| 自動承認 ON + スコア > 閾値 | 手動審査キューへ | `pending` |
| 自動承認 ON + スコア ≤ 閾値 | **自動承認** | `approved` |

### 2.4 提出履歴の確認

- 自分の提出一覧が表示される
- 各提出の状態（pending / approved / rejected）を確認できる
- `pending` 状態の提出は削除可能（審査開始後は削除不可）

## 3. 広報部の運用

### 3.1 ログイン

1. Google OAuth でログイン
2. パスワードモーダルで [広報部] を選択
3. 広報部パスワードを入力 → `josenai_review_koho` 権限が付与される

### 3.2 審査

1. ダッシュボード画面を開く
2. `koho`（広報部宛）の提出物のみが表示される
3. 各提出物の詳細を確認:
   - AI リスクスコアと指摘内容
   - アップロードされたファイル
   - 提出者のコメント（高リスク時の理由等）
4. 審査結果を選択:
   - **承認**: コメントを付けて承認 → status = `approved`
   - **却下**: 却下理由を記入して却下 → status = `rejected`

### 3.3 その他

- 広報部員自身も SNS 素材の提出が可能
- 自分が過去に審査した提出物は引き続き閲覧可能（監査証跡）

## 6. AI判定の詳細

Gemini API がアップロードされた情宣素材を分析し、`josenai_check_items` テーブルの **19 項目** をチェックする。項目は 3 カテゴリに分かれ、重大度に応じたリスク重みが設定されている。

> **注意**: AI判定は情宣素材のリスク要因を機械的に検出する補助ツールであり、法的助言や著作権判断を提供するものではありません。AI の出力は参考情報です。禁止事項・著作権・フォーマットに関する最終判断は、必ず人間の審査者が行ってください。

### 6.1 禁止事項（8 項目 / リスク重み: 30 点）

最も重大な違反。1 項目でも該当すると高リスクとなる。

| コード | 項目名 | チェック内容 |
|--------|--------|------------|
| PRH001 | 反社会的表現 | 暴力、差別、ヘイトスピーチ等の表現 |
| PRH002 | 法令違反 | 法律に抵触する可能性のある表現 |
| PRH003 | 公序良俗違反 | 公序良俗に反する表現 |
| PRH004 | 虚偽情報 | 事実と異なる誤解を招く情報 |
| PRH005 | 個人情報露出 | 許可なく個人を特定できる情報 |
| PRH006 | 大学規則違反 | 近畿大学の規則に違反する内容 |
| PRH007 | 競合他社宣伝 | 企業スポンサー以外の商業宣伝 |
| PRH008 | 政治的主張 | 特定の政治的立場を支持する表現 |

### 6.2 著作権関連（6 項目 / リスク重み: 15 点）

中程度の懸念。ライセンスや許諾で解消可能な場合が多い。

| コード | 項目名 | チェック内容 |
|--------|--------|------------|
| CPR001 | 画像著作権 | 使用画像の著作権処理 |
| CPR002 | フォント著作権 | 商用利用可能なフォントか |
| CPR003 | 音楽著作権 | 動画使用音楽の著作権 |
| CPR004 | キャラクター使用 | 他者キャラクターの無断使用 |
| CPR005 | ロゴ・商標 | 他社ロゴ・商標の無断使用 |
| CPR006 | 二次創作 | ガイドライン準拠の確認 |

### 6.3 フォーマット関連（5 項目 / リスク重み: 5 点）

軽微な指摘。修正が容易な形式的不備。

| コード | 項目名 | チェック内容 |
|--------|--------|------------|
| FMT001 | 必須情報記載 | 日時・場所・団体名等の必須情報 |
| FMT002 | 大学祭ロゴ | 生駒祭公式ロゴの適切な配置 |
| FMT003 | 文字可読性 | 文字サイズ・色の適切さ |
| FMT004 | 画像解像度 | 印刷耐性（300dpi 以上推奨） |
| FMT005 | ファイル形式 | 指定ファイル形式・サイズ |

### 6.4 リスクスコアの算出

- 各項目に対して AI が「該当あり」と判定 → その項目のリスク重みが加算される
- 合計スコアを 0–100% に正規化する
- 例: 禁止事項 1 件 (30) + フォーマット 2 件 (5×2=10) → 40/100 = **40%**

| スコア帯 | リスクレベル | 提出時の動作 |
|---------|-------------|-------------|
| 0–10% | 低リスク | ワンタップで提出完了 |
| 11–50% | 中リスク | 警告表示、確認後に提出 |
| 51–100% | 高リスク | 理由入力が必須 |

## 7. 審査・却下の運用

### 7.0 自動承認された提出物の扱い

自動承認が有効な場合、条件を満たす提出物は審査者の確認なしに `approved` となる。

**ダッシュボードでの表示:**
- 自動承認された提出物には「自動承認」バッジが表示される
- 識別方法: `status = 'approved'` かつ `reviewed_by = NULL`
- フィルタで「自動承認のみ」「手動承認のみ」を切り替え可能

**事後確認の方針:**
- 管理者は自動承認された提出物を定期的にサンプリング確認することを推奨
- 確認頻度の目安: 1日1回、または提出が集中する時間帯の後
- 問題が発見された場合は、通常の審査フローと同様に `rejected` に変更し、却下理由を記入する

**運用推奨事項:**
- 祭り直前は閾値を低めに設定する（5% 推奨）
- 新たなルール変更直後は自動承認を一時的に無効化し、AI 判定の傾向を確認する
- 自動承認率が高すぎる場合は閾値を下げることを検討する

### 7.1 審査の基本方針

- AI 判定はあくまで**参考情報**であり、最終判断は人間の審査者が行う
- AI が見逃した問題を人間が指摘することも、AI が過剰に検出した項目を人間が問題なしと判断することもある
- 審査結果は `reviewer_comment` に記録される（自由記述）

### 7.2 複数審査者の同時操作

- 同一提出物を複数の審査者が同時に審査した場合、後から保存した人に「競合が検出されました」と警告が表示される
- 競合発生時はページを再読み込みし、最新の状態を確認のうえ再審査する
- この仕組みは josenai_submissions テーブルの version カラムによる楽観的ロックで実現される

### 7.3 却下理由テンプレート

却下時は以下のカテゴリを参考に、具体的な修正指示を記載する。

| カテゴリ | 却下理由の記載例 |
|---------|----------------|
| 禁止事項抵触 | 「○○の表現が反社会的表現に該当する可能性があります。該当箇所を修正してください」 |
| 著作権問題 | 「使用されている画像の著作権が確認できません。フリー素材の利用規約またはライセンス証明を添えて再提出してください」 |
| フォーマット不備 | 「生駒祭公式ロゴが配置されていません。提出ガイドを確認のうえ、ロゴを追加して再提出してください」 |
| 情報不足 | 「開催日時・場所の記載がありません」 |
| 解像度不足 | 「印刷に必要な解像度（300dpi 以上）を満たしていません。高解像度版を再提出してください」 |
| その他 | 自由記述 |

### 7.4 却下後の再提出

- 却下された提出者には status = `rejected` と `reviewer_comment` が通知される
- 提出者はコメント内容に従い修正のうえ、新規提出として再提出する
- 現状、再提出と元の提出の自動紐づけ機能はない（将来課題。セクション 12 参照）
- 運用回避策: 再提出時のコメント欄に「PRJ001 の再提出」等と記載する

### 7.5 自動承認の運用設定

#### 閾値設定の推奨手順

1. 初期段階は `auto_approve_enabled = false`（手動審査のみ）で運用開始する
2. 審査実績が蓄積されたら、AI リスクスコアの分布を確認する
3. 低リスク帯の提出物が多い場合、自動承認の有効化を検討する
4. まず閾値 5% で有効化し、自動承認された提出物の品質を確認する
5. 問題がなければ閾値を段階的に引き上げる

#### 閾値別の目安

| 閾値 | 自動承認される範囲 | 推奨シーン |
|------|-------------------|-----------|
| 5% | ほぼ完全に問題のない提出物のみ | 厳格運用・初期導入時 |
| 10% | 低リスク帯（フォーマット軽微指摘のみ） | 標準運用（デフォルト） |
| 20% | フォーマット指摘数件まで許容 | 審査負荷の軽減が必要な場合 |
| 50% | 禁止事項該当以外はほぼ自動承認 | 大量提出時・祭り当日の緊急対応 |

> **注意**: 閾値 50% 以上は禁止事項に該当しない提出物の大半が自動承認されるため、慎重に運用すること。

## 8. AI判定エラー時の対応

| 状況 | 動作 |
|------|------|
| 30秒タイムアウト | 「AI判定をスキップ」ボタンが表示される |
| Gemini API エラー | 「AI判定をスキップ」ボタンが表示される |

**スキップ時:**
- `ai_risk_score = NULL` で提出される
- `ai_risk_details = { "skipped": true, "reason": "timeout" or "api_error" }`
- 管理者のダッシュボードに手動審査対象として表示される

## 9. セッション管理

- ブラウザを閉じるとセッションがリセットされる
- 次回ログイン時に再度パスワード入力モーダルが表示される（毎回権限を選択）
- これにより、共有端末での権限の持ち越しを防止する

### 9.5 障害対応

#### サービス障害時の確認手順

| サービス | 確認方法 | 代替手段 |
|---------|---------|---------|
| Supabase | Supabase ダッシュボード → Health | 復旧待ち（提出受付を一時停止） |
| Gemini API | Edge Function ログ確認 | AI判定スキップで運用継続 |
| Google Drive | Edge Function ログ確認 | 提出受付を一時停止 |

#### ログ確認方法

- Supabase Edge Function ログ: ダッシュボード → Edge Functions → Logs
- フロントエンドエラー: ブラウザ DevTools → Console

### 9.6 バックアップと復旧

| データ | バックアップ方法 | 復旧方法 |
|--------|----------------|---------|
| Supabase DB | Supabase 自動バックアップ（Pro プラン: 7日間） | ダッシュボードからリストア |
| Google Drive ファイル | Google Drive のバージョン管理 | ファイル単位で復元 |
| マスタデータ (CSV) | 元 CSV ファイルを保管 | 再インポート |

### 9.7 祭り当日の緊急対応

| 状況 | 対応 | 実行者 |
|------|------|--------|
| AI判定が全件エラー | `ai_timeout_seconds` を短くし、スキップ運用に切替 | 管理者 |
| 提出が殺到し遅延 | Supabase ダッシュボードで負荷を確認 | 管理者 |
| 不適切な提出物の流出 | `submission_enabled = false` で受付停止 | 管理者 |
| 管理者パスワード失念 | Supabase DB の josenai_app_settings を直接更新 | システム管理者 |

## 10. 提出物のライフサイクル

```
提出 → AI判定
         ├─ [自動承認条件充足] → approved（自動承認）
         │    ※ reviewed_by = NULL, reviewer_comment に自動承認の旨を記録
         │
         └─ [自動承認条件不充足 or 自動承認OFF]
              → pending → approved（手動承認）
                        → rejected（却下）→ 修正して再提出

※ pending 状態のみ提出者が削除可能
※ 削除時は Google Drive のファイルも同時に削除される
※ 自動承認された提出物も管理者が事後に rejected に変更可能
```

## 12. 運用改善の検討事項

以下は現在の仕様にはないが、運用上検討すべき将来課題である。

### 12.1 再提出の紐づけ

- **現状**: 却下後の再提出は「新規提出」として別レコードになる
- **改善案**: 「どの提出の修正版か」を関連付けることで、審査者が差分を把握しやすくなる
- **暫定運用**: コメント欄に「PRJ001 の再提出」等と記載

### 12.2 提出締切の管理

- **現状**: `submission_enabled` フラグの ON/OFF のみ
- **改善案**: 媒体種別ごとに締切日時を設定し、段階的な締切管理を可能にする

### 12.3 自動承認の拡張

- **現状**: グローバル単一閾値による自動承認が実装済み
- **改善案**: 部局別（企画管理部/広報部）に異なる閾値を設定可能にする
- **検討事項**: 媒体種別ごとの閾値設定（ポスターは厳格に、SNS画像は緩和等）

### 12.4 審査統計ダッシュボード

- **現状**: なし
- **改善案**: 承認/却下件数、平均審査時間、カテゴリ別集計等を可視化する
